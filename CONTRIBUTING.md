## 🔧 开发环境搭建

### 前置要求

| 依赖         | 版本要求    | 说明                           |
| ------------ | ----------- | ------------------------------ |
| JDK          | 1.8（推荐） | 高版本 JDK 可能出现 HttpServer 权限报错 |
| IDE          | 不限        | 推荐 VS Code、IntelliJ IDEA、Eclipse |
| Git          | 任意版本    | 用于版本控制                   |

### 快速启动

1. **确认 JDK 版本**：
   ```bash
   java -version
   # 确保显示 Java 8（如 1.8.0_xxx）
   ```

2. **导入项目到 IDE**：
   - 将项目根目录作为普通 Java 项目打开。
   - 确保所有 `.java` 文件位于同一包（默认无包声明）下。

3. **运行方式一——控制台终端**：
   - 运行 `CustomerView.java` 中的 `main` 方法。

4. **运行方式二——网页可视化**：
   - 运行 `HttpServerHandler.java` 中的 `main` 方法。
   - 浏览器访问 [http://localhost:8080](http://localhost:8080)。

> **注意**：控制台与网页版本不可同时运行，两者数据相互独立。

## 📐 代码规范

本项目为纯原生 JDK 实现，**无第三方框架依赖**。请遵守以下规范：

### 通用

- 源文件编码使用 **UTF-8**（IDE 默认即可）。
- 缩进使用 **Tab**（保持项目现有风格一致）。
- 每行长度建议不超过 **120 个字符**。
- 类名使用 **大驼峰命名法**（PascalCase），如 `CustomerList`。
- 方法名和变量名使用 **小驼峰命名法**（camelCase），如 `addNewCustomer`。
- 常量使用全大写 + 下划线，如 `MAX_CAPACITY`。

### 注释

- 类头部使用 `//` 行注释简要说明类职责。
- 复杂方法或算法逻辑需添加注释说明。
- 代码中的 `TODO` 注释需标注贡献者标识，如 `// TODO: ... - @username`。
- 临时调试代码不允许提交到主分支。

### CRUD 操作

- 所有数组操作必须校验下标越界。
- 新增客户前检查数组容量。
- 查询客户时返回副本（`getAllCustomers()` 已实现该模式）。
- 删除客户后记得将尾部置空并更新 `total` 计数。

### 前端（内嵌 HTML/JavaScript）

- HTML 标签保持语义化，CSS 类名使用小写字母 + 连字符 `-`。
- JavaScript 函数名使用小驼峰。
- 页面交互需提供明确的成功/失败反馈提示。

## 🌿 分支管理

| 分支名称      | 用途                         | 基准分支     |
| ------------- | ---------------------------- | ------------ |
| `main`        | 稳定版本，保持可运行状态     | —            |
| `dev`         | 开发主分支，集成新功能       | `main`       |
| `feat/<名称>` | 功能开发分支                 | `dev`        |
| `fix/<名称>`  | Bug 修复分支                 | `dev`        |
| `docs/<名称>` | 文档修改分支                 | `main` 或 `dev` |

示例：

```bash
git checkout -b feat/file-persistence
git checkout -b fix/array-out-of-bounds
git checkout -b docs/contributing-guide
```

## 💬 提交信息规范

提交信息请遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<类型>: <简短描述>

<可选详细说明>
```

### 常用类型

| 类型       | 说明                 |
| ---------- | -------------------- |
| `feat`     | 新功能               |
| `fix`      | Bug 修复             |
| `docs`     | 文档变更             |
| `refactor` | 代码重构（无功能变化） |
| `style`    | 代码格式调整         |
| `test`     | 添加或修改测试       |
| `chore`    | 构建或工具变更       |

### 示例

```
feat: 添加客户文件持久化功能

- 程序退出时将客户数据保存到 customer.dat
- 启动时自动从文件恢复数据
- 文件不存在时静默跳过
```

```
fix: 修复删除客户时数组尾部未置空的问题
```

```
docs: 完善 README 运行教程
```

## 🔄 Pull Request 流程

1. **确保分支名符合规范**，如 `feat/file-persistence`。
2. **本地充分验证**：
   - 无编译错误。
   - 控制台版本与网页版本均可正常运行。
   - 增删改查功能完整有效。
3. **向目标分支提交 PR**：
   - 功能/修复 → `dev`
   - 文档 → `main`
4. **PR 标题**使用与提交信息一致的格式，如 `feat: 添加客户文件持久化功能`。
5. **PR 描述**中请包含：
   - 改动内容概述。
   - 改动原因或动机。
   - 如果涉及 UI 变化，可附截图。
6. **等待 Review**：维护者会在 1-3 个工作日内 Review。
7. **处理反馈**：如有修改意见，直接在对应分支继续提交，PR 会自动更新。
8. **合并**：通过 Review 后由维护者合并到目标分支。

## 🐛 报告 Issue

### Bug 报告

请包含以下信息：

- **环境信息**：操作系统、JDK 版本。
- **运行模式**：控制台版本还是网页版本。
- **复现步骤**：清晰的、从启动到出现问题的完整操作流程。
- **期望行为**：您认为应该发生的结果。
- **实际行为**：实际发生的结果（如有错误信息请完整贴出）。
- **截图/日志**：方便定位问题。

### 功能建议

请描述：

- 要解决什么痛点或场景。
- 期望的功能表现。
- 是否愿意参与实现（欢迎提交 PR！）。
